# 05. 進め方（学習方針・役割分担）

## 目的の再確認
プロダクト完成より**手を動かす実装力の回復**が目的。だからAIに書かせると逆効果。
ただし「いきなり全部手書き」は初心すぎてフリーズするので、縛りは段階調整する。

## ストイック・モード（"見本→真似"方式）
- 中核ロジック（モデル/バリデーション/SQL/API設計/テスト）は**自分の指でタイプ**する。生成コードを丸写ししない。
- コマンドやRailsの定型(idiom)は**教わってよい**（それは「代わりに設計する」のとは違う）。
- 基本パターン: **「Claudeが1つ見本を示す → 同じパターンを次のケースで自分が打つ」**。
  例: Claudeが `Account` の作り方を1個見せる → `JournalEntry` は自分で同じ手順で打つ。

## Claudeの役割
**コードを書く人ではなく、設計レビュアー兼スパーリング相手。**
- 叩き台ロジックを丸ごと出さない。
- 設計の「問い」を投げる。書いたコードをレビューする。詰まったらヒントだけ返す。
- 詰まった時は、コードでなく「問い」を持ってくる（例:「この不変条件はモデルとDBどっちで守るべき？」）。

## トークン戦略（消費を抑える役割分担）
- **指の練習はオフラインで一人**: 公式 [Rails Guides](https://guides.rubyonrails.org/) 等での純粋な反復にClaudeは不要（トークン0）。
- Claudeに来るのは **(a)設計の壁打ち (b)詰まった時の問い (c)コードレビュー** だけ。
- レビューは**まとめて**出す（1行ずつ見せない）。`git diff` を貼れば最小トークン。
- **設計が決まったら会話でなく docs/ に書く**。チャットは揮発、mdは残り次セッションの文脈復元に使える。

## 各イテレーション後
`docs/notes/iter-NN-xxx.md` に3行ずつ:
- 設計判断（何をなぜ選んだか）
- 詰まった点
- 次にやること
